## Required Layout Standard

All invoice output paths, manual and direct-send, should use the same layout rules:

1. Customer NIT must be visible and labeled, for example `NIT: 108207552`.
2. Customer block wording must be corrected consistently. Current manual output uses `FACTURAR A`.
3. Bank/payment information must be removed from invoice PDFs unless explicitly required by a separate invoice type.
4. Manual output and BC direct-send output must use the same report layout/template version.

## Current Conclusion

The currently observed layouts are not correct:

- The BC direct-send screenshot is not correct because the customer NIT is not visible and the wording appears as `FACTRURA A`.
- The manual Jasper PDFs are not fully correct because the customer NIT is visible but unlabeled, and the bank/payment footer still appears.
- The BC API PDF-generation path is not correct because it fails before rendering with `No payment information is provided in Company Information. Review the report.`

The exact BC setup rows that select those layouts could not be read through the currently published API/OData surface. The read-only layout audit API added in extension version `0.1.1.0` is required to identify the exact selected report IDs/layout names in BC.

## Follow-up: Read-Only Layout Audit API

The BC standard/public API did not expose the layout records needed for a definitive automated check. The repo now includes read-only API pages in the existing AL extension to expose the relevant setup after controlled publishing:

- `reportSelections`: global report selections, including invoice email attachment layout.
- `customReportSelections`: customer-specific document layouts.
- `reportLayoutSelections`: default layout selected for each report/company.
- `reportLayouts`: available installed layouts.
- `documentSendingProfiles`: profiles used by Post and Send / direct send.
- `customerLayoutSetup`: customer NIT, email, and selected document sending profile.

These pages are under:

`/api/mtmlogix/layoutAudit/v1.0/companies({company_id})/...`

The COTTONTEXTILE-specific read sequence is:

1. Read `customerLayoutSetup` for `C00081`.
2. Read the customer `documentSendingProfile`, or the default profile if the customer field is blank.
3. Read `customReportSelections` for source number `C00081`; invoice rows here override global sales report selection.
4. Read `reportSelections` for invoice-related usage rows with `useForEmailAttachment = true`.
5. Match the effective `reportId` and layout fields to `reportLayoutSelections` and `reportLayouts`.

Classification rule:

- If the effective BC email attachment layout is different from the corrected manual layout, the BC direct-send setup is not correct.
- If the selected layout renders missing customer NIT, `FACTRURA`, or bank/payment footer, the selected layout is not correct.
- If BC PDF rendering still fails with `No payment information is provided in Company Information`, the selected BC API/direct-render report path is not correct even before visual comparison.
